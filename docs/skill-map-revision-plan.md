# スキルマップ修正方針

> PO（乙部さん）との議論（2026-04-10）に基づく修正方針

## 背景・動機

- 本プロダクトはNEDO技能伝承コンテスト向けのデモ
- 審査員は製造業バックグラウンドの方が中心
- 現状のスキルマップは組織論・コアコンピテンシー寄りの抽象的・定性的な表現が多く、製造業の審査員に伝わりにくい
- 動画撮影（ピッチ審査）での見せ方を強く意識する必要がある

---

## 1. 習熟レベル定義の改訂

### 現状

| Lv | ラベル | 方向性 |
|----|--------|--------|
| 1 | 認知 | トピックに触れたことがある |
| 2 | 理解 | 論点を説明できる |
| 3 | 応用 | 自分で論点を提起できる |
| 4 | 統合 | 他の観点と関連づけて判断できる |

### 修正方針

**「問題解決」軸に再定義する。** 組織論的な表現（「組織リードできる」等）ではなく、ものづくりのQCDE（品質・コスト・納期・環境）に対してどのレベルの問題解決ができるかで段階を表現する。

| Lv | 方向性（案） |
|----|-------------|
| 1 | 基本的な手順を理解し、マニュアルに沿って作業できる |
| 2 | 標準的な問題を自力で特定・解決できる |
| 3 | 非標準的な問題に対して仮説を立て、QCDEの観点から解決策を提案できる |
| 4 | 複数工程・複数観点を横断し、QCDEの最適バランスを判断して問題解決をリードできる |

### 各スキルのレベル記述も同様に修正

- `skillCatalog.ts` の各スキルの `levels` 記述を、問題解決・ものづくりの完成度向上の文脈に書き換える
- 「組織変革のドライバー」「リーダーシップ」等の抽象表現を避け、具体的な製造業の問題解決行動で記述する

### 対象ファイル

- `lib/compath/domain/skillMap/types.ts` — `SKILL_LEVEL_LABELS` の修正
- `lib/compath/domain/skillMap/skillCatalog.ts` — 全スキルのレベル記述の書き換え

## 2. スキルカテゴリの再編

### 現状（11カテゴリ・70項目）

ビジョン・戦略 / マネジメント / 工程設計 / 加工技術 / 品質管理 / 設備・保全 / 材料・素材 / デジタル・IT / 市場・競合 / コミュニケーション / 安全・環境

### 修正方針

技能伝承コンテストの文脈に合わせ、**ものづくり現場の具体的スキル**を中心に再編する。

#### a. カテゴリレベルの変更

| 現状カテゴリ | 変更方針 | 理由 |
|-------------|---------|------|
| **ビジョン・戦略** | **「顧客要求・ビジネス要件」に改称・再定義** | お客様からの要望・部品の用途・ビジネス上の要件を把握できるかという方向に。「事業戦略」「知財戦略」等の経営寄りの項目は削除または統合 |
| **マネジメント** | **「生産管理」に改称・具体化** | チームマネジメントのような広い概念ではなく、生産管理・工程管理など製造業で具体的に必要なスキルに |
| **工程設計** | 維持（そのまま） | POからもOKの評価 |
| **加工技術** | 維持 + 項目追加 | **金型設計・工法開発**を明示的に独立項目化 |
| **品質管理** | 維持 | — |
| **設備・保全** | **「保守・保全」として独立性を強化** | 加工と保守保全は別領域。現場系フィールドスキルとして保守保全の重要性を明示 |
| **材料・素材** | **素材ごとに具体化** | 「金属材料特性」のような丸め方ではなく、鉄・アルミ・ステンレス等の素材別に項目を分ける |
| **デジタル・IT** | **優先度を下げる / 統合** | 技能伝承コンテストの文脈では主役ではない。必要最小限に絞る |
| **市場・競合** | **「顧客要求・ビジネス要件」に統合** | 顧客ニーズ把握等は上記の新カテゴリに吸収 |
| **コミュニケーション** | **優先度を下げる / 統合** | 同上。技能伝承の文脈では加工・保守保全を前面に |
| **安全・環境** | 維持 | — |

#### b. 項目数について

- 現状70項目だが、**もっと増やす**（見栄えを良くするため）
- 特に素材知識（鉄・アルミ・ステンレス・樹脂等）、加工技術（金型成形・工法開発等）の細分化で項目数を増加させる
- **全カテゴリのスキル項目数を同数に揃える**（ヒートマップが全体で綺麗な長方形になるようにする）

### 対象ファイル

- `lib/compath/domain/skillMap/skillCatalog.ts` — `SKILL_CATEGORIES` と `SKILL_DEFINITIONS` の全面改訂
- `lib/compath/infrastructure/repositories/skillMapSeedData.ts` — サンプルデータの再作成（新カテゴリ・新スキルIDに合わせる）

## 3. UIレイアウトの改訂

### 修正方針: 一画面完結

**最重要要件: スクロール・タブ切り替えなしで一画面に全情報を表示する。**

- ピッチ審査では画面を動かされながら見るのは審査員にとって非常に疲れる
- 一画面で全体像がじっくり見られることが重要
- 動画撮影時に「一画面固定」で説明できることがベスト

### 具体的な変更

#### a. タブ構成の廃止 → 一画面統合

現状の3タブ（概要 / スキル習熟度 / 成長曲線）を廃止し、**一画面にヒートマップ + レーダーチャートを同時表示**する。

#### b. レイアウト案

```
┌─────────────────────────────────────────────────────┐
│ [ヘッダー: ユーザー名 / 役割]                          │
├───────────────────────────────────┬─────────────────┤
│                                   │  レーダーチャート   │
│   スキル習熟度ヒートマップ           │  (五角形/六角形)   │
│   （メイン領域）                    │                  │
│                                   │  サマリー情報      │
│                                   │                  │
├───────────────────────────────────┴─────────────────┤
│ (必要に応じて成長曲線を下部に配置)                       │
└─────────────────────────────────────────────────────┘
```

- ヒートマップ右側の空白にレーダーチャート（QCDESの五角形インジケーター）を配置
- 全画面表示時に空白が出ないようにする

#### c. ヒートマップのセルサイズ

- 動画撮影時に文字が読めるサイズを確保
- 項目数が増えても可読性を維持するサイズ調整
- 現状 56x48px — 項目数増加に合わせて調整が必要

### 対象ファイル

- `components/skill-map/SkillMapDashboard.tsx` — タブ構成廃止、一画面レイアウトに改修
- `components/skill-map/SkillHeatmap.tsx` — セルサイズ・レイアウト調整
- `components/skill-map/QCDESRadar.tsx` — ヒートマップ横への配置調整

## 4. 対象外（今回は変更しない）

- 思考品質スコアの算出ロジック（`assessSession.ts` / `skillProfile.ts`）
- API エンドポイントの構造
- 成長曲線（GrowthTimeline）のロジック — レイアウト配置のみ変更

| 優先度 | 項目 | 理由 |
|--------|------|------|
